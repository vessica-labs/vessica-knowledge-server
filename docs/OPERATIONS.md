# Vessica Knowledge Server Operations

## Purpose

The knowledge server is the hosted authority for a Vessica workspace. Humans and coding agents normally use `ves`; direct HTTP access is intended for the Vessica control plane, cloud sandboxes, diagnostics, and future integrations.

## Runtime modes

- Embedded solo mode is compiled into `ves`, uses SQLite and FTS5/BM25, and requires no server or embedding credential.
- Hosted mode runs this service with the isolated knowledge database and pgvector. An embedding provider is optional; hosted writes never fall back to SQLite.

## Production image

Release images are public at:

```text
ghcr.io/vessica-labs/vessica-knowledge-server:v<version>
```

Deploy the immutable digest resolved from the release tag. `ves up` performs this resolution and records both the semantic version and digest.

## Required environment

| Variable | Required | Purpose |
|---|---:|---|
| `VES_KNOWLEDGE_DATABASE_URL` | Hosted | URL for the isolated `vessica_knowledge` database; enables hosted mode |
| `KNOWLEDGE_API_TOKEN` | Hosted | Ordinary workspace read/write token |
| `KNOWLEDGE_EXPORT_TOKEN` | Hosted | Separately scoped snapshot export/import token |
| `KNOWLEDGE_WORKSPACE_ID` | Hosted | The only workspace accepted by this service instance |
| `EMBEDDING_API_KEY` | Optional | User-funded embedding provider credential; omit for lexical retrieval |
| `EMBEDDING_BASE_URL` | No | OpenAI-compatible embeddings endpoint |
| `EMBEDDING_MODEL` | No | Defaults to `text-embedding-3-small` |
| `RERANK_ENABLED` | No | Must be `true` to allow conditional reranking; defaults disabled |
| `RERANK_API_KEY` | Optional | Separately managed Responses API credential; required only when reranking is enabled |
| `RERANK_BASE_URL` | No | Optional OpenAI-compatible Responses API endpoint |
| `RERANK_MODEL` | No | Defaults to `gpt-5.6-luna` |
| `PORT` | No | HTTP port, default `8080` |
| `KNOWLEDGE_SQLITE_PATH` | Solo server debugging | SQLite file when `VES_KNOWLEDGE_DATABASE_URL` is absent |

Never put secrets in image layers, repository files, CLI arguments generated by agents, or sandbox variables. Use Railway secrets and `ves` credential storage.

## Startup and readiness

Infrastructure provisioning creates `vessica_knowledge`, grants its dedicated role, and enables pgvector before deploying this service. Startup verifies pgvector, then applies the independent `knowledge_schema_migrations` history. It never reads or writes the control plane's `vessica_control` database or `schema_migrations` table. Hosted startup fails if the extension, database URL, or required service credential is absent.

```bash
curl -fsS https://<endpoint>/healthz
curl -fsS https://<endpoint>/readyz
```

`/readyz` distinguishes API, database, migrations, embedding worker, and index state. `index: catching_up` is healthy after imports or bursts of writes; lexical retrieval remains available while embeddings are pending.

## API contract

Use `openapi.yaml` as the HTTP source of truth. Successful and failed domain responses use the `vessica.knowledge/v1` envelope. Every mutation requires:

```text
Authorization: Bearer <token>
Idempotency-Key: <stable retry key>
X-Actor-ID: <actor>
X-Actor-Type: human|user|service|agent
X-Provenance-Source: <source>
```

The service rejects workspace IDs other than `KNOWLEDGE_WORKSPACE_ID`. Export and import require `KNOWLEDGE_EXPORT_TOKEN`.

## Retrieval

`POST /v1/memories:retrieve` is the shared retrieval-v2 API. It returns ranked memories plus component explanations, retrieval mode, index freshness, ambiguity state, and reranker metadata. `GET /v1/memories?q=` remains the compatibility and administrative lexical-search endpoint.

Retrieval v2 obtains at most 50 lexical and 50 semantic candidates after workspace, scope, current-version, active-state, and temporal filters. Weighted reciprocal-rank fusion combines the sources; entity and scope specificity are strong signals, while importance and confidence are tie-breakers. Episode recency never displaces durable instructions or decisions. Ordering ends with version, update time, and ID for reproducibility.

Context responses expose `ranking.version`, component weights, deterministic artifact policy, context order, omissions, and the instruction override rule. Explicit artifact IDs and versions are authoritative. Ordinary artifacts must be lexically or entity relevant; unselected artifacts and episodes have separate type budgets so durable memory retains context capacity.

- Solo mode reports `retrieval_mode: lexical`.
- Hosted mode reports `retrieval_mode: lexical` and `embedding_state: not_configured` when no key is supplied. This is healthy.
- Hosted mode reports `retrieval_mode: semantic_hybrid` when an embedder is configured; lexical retrieval remains available while a backfill catches up.
- Pending or failed embeddings do not hide memory from lexical retrieval.

Conditional reranking receives at most 12 candidates that already passed authorization and lifecycle filters. It can reorder or reject IDs but cannot add IDs. It is skipped for fewer than three candidates and confident exact matches, times out after 2.5 seconds, and falls back to deterministic hybrid order on all provider or schema failures. Logs and errors must contain metadata only, never memory text, raw model output, or credentials.

## Embedding jobs

Embedding jobs are durable database records. Writes commit the memory, event, projection, and pending job before background processing. Jobs retry with bounded backoff and become `failed` after five attempts. A later immutable memory version creates a new job without changing prior vectors.

## Promotion and recovery

Use `ves railway up`, not direct import, for normal promotion. It:

1. Freezes the local authority.
2. Exports an ordered, checksummed snapshot.
3. Imports idempotently while preserving IDs and versions.
4. Verifies counts, event watermark, artifact hashes, memories, entities, and relationships.
5. Runs a representative context request.
6. Saves a read-only SQLite recovery copy.
7. Switches configuration to hosted only after verification.

If promotion fails, local SQLite remains authoritative. Rerun the same command. Do not manually set `knowledge.mode: hosted` and do not create a writable local fallback after promotion.

Import progress is retryable but process-local status is intentionally an MVP limitation; durable chunk checkpoints are tracked in the CLI follow-up document.

## Troubleshooting

### Service will not become ready

Inspect Railway deployment state and logs. Common causes are a missing export token, an invalid knowledge-database URL, or a `vessica_knowledge` database that was not initialized with pgvector. A missing embedding key is a healthy lexical configuration.

### Hosted CLI returns unauthorized

Run `ves auth status --json`, reconnect the workspace, or rerun `ves railway up`. Do not copy bearer tokens into plugin files.

### Context is lexical in hosted mode

Check the embedding provider credential and `/readyz`. A provider outage deliberately degrades the request to lexical retrieval.

### Index is catching up

Wait for durable jobs to drain and recheck `/readyz`. Knowledge remains queryable during catch-up.

### Promotion verification fails

Keep SQLite authoritative. Compare `ves knowledge export --json` counts and watermark with the hosted export, inspect service logs, then retry with the same snapshot.
